## Summary

本文件约束首个可开发版本的边界，避免在进入编码前就把需求扩散到备份、统计、相册导入等非核心方向。

## MVP Must Have

- Android 端 Flutter 应用基础壳。
- 首版只保留首页和设置页两个主页面。
- 每个自然日只允许保存一条 OOTD 记录。
- 每条 OOTD 记录固定为 `1` 张主图和最多 `3` 张细节图。
- 创建 OOTD 时可拍照，也可从系统相册选择图片。
- 拍后基础裁剪。
- 图片压缩并落地到 App 私有目录。
- 本地数据库记录一条 OOTD。
- 本地数据库记录该 OOTD 下的多张照片。
- 首页查看最近记录和历史记录。
- 首页根据标签筛选 OOTD。
- 首页默认只显示“喜欢”的 OOTD 主图。
- 支持将 OOTD 标记为“不喜欢”，并从首页灵感库移出。
- 设置页查看“不喜欢”的 OOTD，并支持恢复。
- 设置页创建、编辑、重命名、删除标签。
- 标签与备注录入。
- 首页内进入独立全屏详情，查看该记录的多张照片与元数据。
- 支持显式永久删除记录。

## MVP Should Have

- 缩略图预生成。
- 空状态。
- 简单错误提示，例如保存失败、目录写入失败。

## MVP Won't Have

- 云同步。
- 登录系统。
- 相册批量导入。
- 日期范围筛选。
- AI 自动识别穿搭风格。
- 自动获取气温。
- 保存到系统相册。
- 图像美颜或复杂滤镜。
- 分享到社交平台。
- Web 或桌面版。

## Preferred UX Direction

- 首页优先展示“快速拍照 + 标签筛选 + 历史记录”，而不是复杂的统计面板。
- 浏览页优先采用性能更稳的固定比例网格。
- 首页默认只承担“找灵感”职责，因此只展示“喜欢”的 OOTD。
- 设置页负责管理标签和“不喜欢”的 OOTD，不在首页直接创建标签。
- 首页使用设置页创建好的标签作为筛选入口。
- 详情页使用独立全屏展示同一条 OOTD 的多角度照片，首页只承担主图浏览职责。

## Exit Criteria

达到以下条件后，可认为 MVP 设计层面已准备好进入正式编码：

- 产品范围无明显歧义。
- 数据模型已确认。
- 存储策略已确认。
- 架构分层已确认。
- 当前计划中的第 0 阶段任务全部转为 confirmed。
